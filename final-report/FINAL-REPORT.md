# Tech Warm-Up II: Final Report

## The Process & Data:

We built our Data Heist slot machine in 14 phases by breaking the architecture down into 46 prompts rather than trying to "one-shot" the entire application with just one. Initial stages started with building and isolating the core mathematical function and foundation of the game. This included state management and fair RNG, which we focused on before any UI code was implemented. We also adhered to test-driven development by writing Jest validation suites for our payout rules. Keeping the prompts small and focused prevented AI hallucinations and ensured that the foundation of the game mechanics were mathematically verifiable early in the pipeline. 

The implementation of the core state loop and software-defined RNG was guided by insights into modern slot architectures that our team did research on. To meet industry standards for Return to Player and volatility, we engineered our reel strips to exactly 64 stops, weighting symbols with the goal of balancing the psychological loop of frequent, small wins against rare, high-value jackpots. Furthermore, our research into player retention and sensory feedback drove the broad feature set of our game including: tiered win animations, transparent payline highlights, and the Neural Chip scatter bonus. Enforcing these specific and research-backed mechanics against the AI's generated code is what exposed the need for rigorous hands-on review.

Once the initial logic was generated, our workflow shifted to continuous integration and auditing with a human in the loop. We were able to achieve consistent styling using ESLint and htmlhint right at Phase 0 in order to catch syntax issues and stop technical debt from accumulating. The bulk of the later phases involved manual logic audits to fix critical architectural flaws created by AI during generation. Some of these flaws included a 25x payout bug, a double-counting spin state, and a percentage-based payout formula that we have to rewrite into a standard fixed multiplier table. By treating the AI as a collaborative refactoring tool rather than being totally hands-off with it, we were able to produce a stable and maintainable backend verified by passing 90 unit tests and 8 Playwright scenarios.

## Results & Discussion:

After engineering with AI, many of the results throughout the process point to a common frustration of token limits, which caused many more issues than expected. Due to the nature of the assignment wanting a more refined product, it was likely better to have one individual run Claude Code with specific prompts in order to maintain better context and continuity. However, due to the token limit, one person alone was not able to generate a well refined product before running out of tokens. This meant that there would be a “pass-off” to another team member who would need to use their own tokens in order to continue the project.

This caused a lot of issues, firstly being the deletion of context that one of our group members experienced when running out of tokens, causing a lot of hard work to be thrown down the drain. Secondly, when passing the baton to a fellow team member, Claude Code would not pass over the context that one member had to the next, causing some discrepancies between the knowledge that Claude Code had between members.

To add to this, trying to limit token usage was a difficult task, since there were many tokens that would be eaten up as a result of the AI’s results. Oftentimes the AI would generate an issue that needed to be fixed, which would require a new prompt to respond to that issue, eating up more tokens and forcing more pass-offs between members. In addition, as the project grew from the first prompts, there would naturally be more code that was added into the final product, which meant that Claude Code would need to scan through the entire code base, eating up precious tokens that contributed to the issue.

In conclusion, engineering with AI can be a frustrating experience especially due to the limits that are put into place, and generally is not a “savior” tool that can magically create outstanding projects and programs. We as a group have more negative feelings towards allowing AI to create and polish its own product. As much as AI is capable of creating a “working” product, understanding the product that AI creates is a difficult task due to the “randomness” of it all. Overall, we feel it is still important to understand programming and other parts of design ourselves.
